# SenRi FFI 自定义后端功能实现文档

---

## 1. 概述

本功能允许用户注入完全自定义的 FFI 后端实现，替换 SenRi 内置的运行时自动检测机制。

用户只需实现一个接口（`LibraryLike`），SenRi 内核通过适配器包装器将其接入现有架构。**用户负责底层 FFI 引擎的所有细节（加载、绑定、内存分配、回调、错误信息），SenRi 继续提供上层统一 API（类型系统、Pointer、struct、alloc/free 等）**。

本功能不会影响默认行为：不传 `backend` 时，SenRi 仍使用内置适配器自动检测运行时。

## 3. 用户接口定义（LibraryLike）

以下为强制/可选方法完整契约，包含详细的参数、返回值及语义约束。

```typescript
// 文件：src/types/library-like.ts

import { NormalizedType } from './normalized';

/**
 * 用户自定义后端必须实现的接口。
 * 除标记为 optional 的方法外，其余均为强制实现。
 */
export interface LibraryLike {
  // ---------- 生命周期钩子（可选） ----------
  /**
   * 可选：后端初始化函数。在包装器构造时调用一次。
   * 适合用于初始化线程池、全局驱动等。
   * 若抛出异常，包装器会将其转换为 FFIError 并中止加载。
   */
  init?(): void;

  /**
   * 可选：后端销毁函数。在全局适配器被替换或进程退出前调用。
   * 注意：此函数必须为同步操作，不可抛出异常（框架会静默吞噬异常）。
   */
  destroy?(): void;

  // ---------- 库操作（强制） ----------
  /**
   * 打开动态库。
   * @param path 库文件路径（绝对或相对）
   * @returns 不透明句柄，可以是任意类型（number、object、symbol等），
   *          后续会原样传给 bind/close。
   * @throws 若加载失败，应抛出错误（SenRi 会包装为 FFIError）。
   */
  open(path: string): any;

  /**
   * 绑定同步 C 函数。
   * @param handle 由 open 返回的句柄
   * @param name 函数符号名
   * @param retType 标准化后的返回类型（NormalizedType）
   * @param argTypes 标准化后的参数类型数组
   * @returns 一个可调用的 JavaScript 函数，该函数接收与 argTypes 匹配的 JS 值，
   *          返回与 retType 匹配的 JS 值（如 bigint、number、string、Pointer 等）
   * @throws 若绑定失败（符号不存在、类型不匹配等），应抛出错误。
   */
  bind(
    handle: any,
    name: string,
    retType: NormalizedType,
    argTypes: NormalizedType[]
  ): Function;

  /**
   * 关闭库，释放资源。
   * @param handle 由 open 返回的句柄
   * 注意：即使关闭失败，也不应抛出异常，静默处理。
   */
  close(handle: any): void;

  // ---------- 内存管理（强制） ----------
  /**
   * 分配原生内存。
   * @param size 字节数（正数）
   * @returns 对象必须包含三个字段：
   *   - __ptr: bigint  原生内存地址
   *   - __buf: any     可供 DataView 读写的缓冲区（通常为 Buffer、Uint8Array 或 ArrayBuffer）
   *   - __size: number 分配的大小
   * @throws 分配失败应抛出错误。
   */
  alloc(size: number): {
    __ptr: bigint;
    __buf: any;
    __size: number;
  };

  /**
   * 释放之前由 alloc 分配的内存。
   * @param ptr 由 alloc 返回的对象
   * 静默处理，不可抛出异常。
   */
  free(ptr: any): void;

  /**
   * 获取 ArrayBuffer 或 TypedArray 的原生起始地址。
   * @param buffer 任意 ArrayBuffer 或 TypedArray
   * @returns bigint 地址
   * @throws 若无法获取地址，应抛出错误。
   */
  addressOf(buffer: ArrayBuffer | ArrayBufferView): bigint;

  // ---------- 回调管理（强制） ----------
  /**
   * 将 JavaScript 函数注册为 C 回调函数。
   * @param func 被调用的 JS 函数
   * @param retType 返回值类型（NormalizedType）
   * @param argTypes 参数类型数组
   * @returns 对象必须包含两个字段：
   *   - __ptr: bigint  C 函数指针（回调地址）
   *   - __cb: any      引擎侧的回调句柄，用于后续释放
   * @throws 注册失败应抛出错误。
   */
  registerCallback(
    func: Function,
    retType: NormalizedType,
    argTypes: NormalizedType[]
  ): {
    __ptr: bigint;
    __cb: any;
  };

  /**
   * 释放先前注册的回调。
   * @param ptr 由 registerCallback 返回的对象
   * 静默处理，不可抛出异常。
   */
  unregisterCallback(ptr: any): void;

  // ---------- 错误信息（强制） ----------
  /**
   * 获取最后一次系统调用的错误码（errno）。
   * @returns 错误码（number），若无错误则返回 0。
   */
  getErrno(): number;

  /**
   * 根据错误码获取可读的错误描述字符串。
   * @param errno 错误码
   * @returns 描述字符串
   */
  getStrerror(errno: number): string;

  // ---------- 异步绑定（可选） ----------
  /**
   * 绑定异步 C 函数（非阻塞），返回 Promise 风格函数。
   * 此方法为可选，若未实现，`Library.funcAsync` 将自动降级为同步包装。
   * @param handle open 返回的句柄
   * @param name 函数名
   * @param retType 返回类型
   * @param argTypes 参数类型
   * @returns 一个函数，该函数返回 Promise
   */
  bindAsync?(
    handle: any,
    name: string,
    retType: NormalizedType,
    argTypes: NormalizedType[]
  ): (...args: any[]) => Promise<any>;
}

// ---------- 辅助类型守卫 ----------
/**
 * 检查对象是否实现了 LibraryLike 接口的所有强制方法。
 * 仅检查方法名存在性，签名验证由构造阶段的烟雾测试完成。
 */
export function isLibraryLike(obj: any): obj is LibraryLike;

/**
 * 返回缺失的强制方法名列表（若全部实现则返回空数组）。
 */
export function getMissingMethods(obj: any): string[];

/**
 * 部分后端类型：允许用户只实现部分方法，其余回退到内置适配器。
 */
export type PartialLibraryLike = Partial<LibraryLike>;

/**
 * 组合工厂：用部分实现 + 内置适配器创建完整后端。
 * 返回的对象满足 LibraryLike 接口，每个方法将优先调用 partial 中对应实现，
 * 若未提供则调用内置适配器的对应方法。
 */
export function createBackendWithFallback(
  partial: PartialLibraryLike,
  builtinAdapter: FFIAdapter
): LibraryLike;
```

## 4. 核心实现机制

### 4.1 Library.load 扩展流程（详细步骤）

`Library.load` 新增第二个参数 `backend`，类型为 `LibraryLike | { new (path: string): LibraryLike }`。

**处理步骤**：

1. **解析 `backend` 参数**  
   - 若 `backend` 为 `undefined` 或 `null` → 使用 `detectRuntime()` 创建内置适配器（原有逻辑）。  
   - 若 `backend` 为 `function`（可构造）→ 调用 `new backend(path)` 获取实例。  
   - 若 `backend` 为对象 → 直接作为实例使用。  
   - 其他情况 → 抛出 `FFITypeError`。

2. **验证实例完整性**  
   - 调用 `isLibraryLike(instance)`，该函数会检查实例上是否存在所有强制方法（`open`、`bind`、`close`、`alloc`、`free`、`addressOf`、`registerCallback`、`unregisterCallback`、`getErrno`、`getStrerror`）。  
   - 若有缺失，调用 `getMissingMethods` 获取缺失列表，并抛出 `FFIError`，消息中包含“缺失方法：a, b, c”。

3. **执行烟雾测试（契约验证）**  
   - 调用 `alloc(1)` 并验证返回对象是否包含 `__ptr`（bigint）、`__buf`（非空）、`__size`（number），然后立即 `free` 释放。  
   - 调用 `getErrno()` 并验证返回类型为 `number`。  
   - 任何验证失败均抛出 `FFITypeError`，明确指出违反的契约条款。

4. **创建适配器包装器**  
   - 实例化 `CustomAdapterWrapper`，传入用户实例。  
   - 该包装器内部持有用户实例引用，并实现 `FFIAdapter` 接口。  
   - 若用户提供了 `init` 方法，此时调用之。

5. **检查资源冲突并更新全局适配器**  
   - 查询全局资源注册表（`ResourceRegistry`）。  
   - 若有活跃资源（之前适配器分配的未释放内存或回调），默认抛出 `FFIError` 阻止切换。  
   - 用户可通过 `force: true` 选项强行切换，此时框架会输出红色警告日志，并清空资源注册表（放弃追踪）。  
   - 调用 `setGlobalAdapter(wrapper)`，更新全局适配器指针。  
   - 若旧适配器持有用户后端且存在 `destroy` 方法，此时调用。

6. **加载库并构造 `Library` 实例**  
   - 调用包装器的 `loadLibrary(path)` → 内部转发给用户后端的 `open(path)`，得到句柄。  
   - 调用 `new Library(handle, path, wrapper)`，将句柄、路径和适配器存入实例。  
   - 返回该 `Library` 实例。

7. **异常处理**  
   - 用户后端的 `open` 或 `loadLibrary` 抛出的任何异常都会被包装为 `FFIError`，并附带原始错误信息（保留 `cause` 字段）。

---

### 4.2 适配器包装器（CustomAdapterWrapper）的内部映射

`CustomAdapterWrapper` 实现 `FFIAdapter` 接口，每个方法委托给用户实例，但执行以下预处理/后处理：

- **类型映射方法（`mapType`、`createPointerType`、`createArrayType`、`createStructType`）**  
  SenRi 内核在缓存函数签名时会调用这些方法。用户后端直接接收 `NormalizedType`，无需转换。包装器实现这些方法时应返回类型的**稳定序列化字符串**（利用内部 `serializeType`），以使内核缓存键不受适配器影响。这些方法不依赖用户后端。

- **所有委托方法均包裹错误转换层**：  
  捕获用户后端的异常，重新抛出为 `FFIError`，并附带上下文（如“加载库失败”、“绑定函数失败”等）。  
  在调试模式（`SENRI_FFI_DEBUG=true`）下，直接透传原始错误，不进行包装，便于调试。

- **`closeLibrary` 和 `free`、`unregisterCallback`**：  
  这类释放资源的方法，即使失败也不应抛出异常，因此包装器内部会静默捕获异常，仅记录日志（若有日志系统）。

- **`getErrno` 和 `getStrerror`**：  
  若用户后端对应方法抛出异常，包装器应返回默认值（`getErrno` 返回 0，`getStrerror` 返回 `"Unknown error"`），避免上层崩溃。

- **资源注册/注销**：  
  `alloc` 在返回前将结果对象注册到 `ResourceRegistry`；`free` 在调用用户后端前先注销该对象。  
  `registerCallback` 在返回前将结果对象注册；`unregisterCallback` 先注销再调用用户后端。

- **`loadLibrary(path)` 和 `closeLibrary(handle)`**：  
  这两个方法是 `FFIAdapter` 的一部分，包装器内部直接调用用户的 `open` 和 `close`。

---

### 4.3 资源注册表与全局适配器安全切换机制

为避免因全局适配器替换导致的内存释放错配，引入中心化的资源追踪器：

- **注册表维护**：  
  注册表是一个内存集合（Set），分别追踪内存块描述对象和回调描述对象。  
  每次 `alloc` 成功时，将返回的对象加入集合；每次 `free` 调用时，将其从集合中移除。  
  同样，`registerCallback` 成功时加入，`unregisterCallback` 时移除。

- **切换检查**：  
  `setGlobalAdapter` 在执行替换前，检查注册表是否为空。  
  若非空，默认抛出异常，阻止切换，并明确提示用户释放所有资源。  
  用户可通过 `force: true` 选项强制切换，此时框架输出醒目错误日志，并清空注册表（放弃追踪）。

- **清理钩子**：  
  在切换前，若旧适配器对应 `destroy` 方法存在，则调用之。  
  在切换后，若新适配器对应 `init` 方法存在，则调用之。

---

### 4.4 内存管理集成（Pointer 与 alloc/free 的协作）

- `alloc(size)` 函数将调用 `getGlobalAdapter().alloc(size)`，得到的对象直接传给 `Pointer` 构造函数。  
- `Pointer` 类内部依赖 `__buf` 进行数据读写，因此用户后端的 `alloc` 必须返回一个包含 `__buf` 的对象，且该 `__buf` 必须能被 `new DataView()` 接受（即 `ArrayBuffer`、`Buffer`（在 Node 中）或 `Uint8Array` 等）。  
- `struct` 的 `toPointer()` 方法会在内部调用 `alloc(structSize)`，并将实例数据写入 `__buf`，然后返回指向该内存的 `Pointer`。  
- 用户后端的 `free` 只负责释放内存，无需关心 `Pointer` 实例的引用计数（由 JS GC 管理）。  
- `Pointer` 实例的 `free` 方法调用会触发全局 `free` 函数，进而调用当前适配器的 `free`，同时资源注册表会确保注销操作。

---

### 4.5 回调集成（FinalizationRegistry 自动回收）

- `callback()` 函数内部：  
  1. 调用 `getGlobalAdapter().registerCallback(func, retType, argTypes)`，获得 `{ __ptr, __cb }`。  
  2. 用 `__ptr` 和 `__cb` 创建 `Pointer` 实例（`Pointer` 会存储 `__cb` 以供释放）。  
  3. 若全局支持 `FinalizationRegistry`（所有目标运行时均支持），则创建一个注册表，将 `Pointer` 实例作为注册值，并在清理回调中调用 `getGlobalAdapter().unregisterCallback(ptr)`，其中 `ptr` 为之前存储的回调对象。  
- 用户后端的 `unregisterCallback` 应负责释放底层 C 回调指针，避免内存泄漏。

---

### 4.6 异步绑定集成（funcAsync 的选择逻辑）

`Library.funcAsync` 的实现需区分运行时和用户后端：

- **若用户后端提供了 `bindAsync` 方法**：  
  在 `funcAsync` 中检查 `this._adapter` 是否为 `CustomAdapterWrapper`，若是，则尝试调用 `backend.bindAsync(handle, name, retType, argTypes)`。若成功，直接缓存返回的异步函数；若抛出异常，则向上传播。

- **若用户后端未提供 `bindAsync`**（或包装器检测到该方法不存在）：  
  - 调用 `this.func(name, retType, argTypes)` 获取同步函数。  
  - 构造一个包装函数，该函数返回一个 `Promise`，内部 `resolve(syncFn(...args))`（或 `reject(e)`）。  
  - 此包装会在事件循环的下一 tick 执行（实际上 `Promise` 会立即 resolve，但同步函数本身会阻塞调用线程）。  
  - **警告机制**：  
    - 检查环境变量 `SENRI_FFI_QUIET`，若未设置（或非 `'1'`/`'true'`），则通过 `console.warn` 输出一次警告，提示“funcAsync 降级为同步调用，可能阻塞事件循环”。  
    - 在返回的函数上附加属性 `_isSyncFallback = true`，以便用户程序检测是否真的异步。  
  - 警告只输出一次（通过一个模块级标志位控制）。

- **对于内置适配器**（非用户后端），保持原有逻辑（Node 用 Worker、Bun 用同步包装、Deno 用原生异步等）。

---

### 4.7 错误处理与类型安全

- 所有用户后端的异常在包装器层都会被捕获并转换为 `FFIError`，同时保留原始错误消息。  
- 在 `Library.load` 阶段，若用户传入的 `backend` 不满足接口要求（类型错误或缺失方法），会抛出 `FFITypeError`（继承自 `FFIError`），提供清晰的错误信息。  
- 若用户后端的 `mapType` 等方法被意外调用（理论上不会），包装器会抛出 `FFIError` 提示“内部错误：不应调用”。

---

### 4.8 与现有内置适配器的共存

- 内置适配器完全保留，检测逻辑不变。  
- 当用户不传 `backend` 时，`detectRuntime()` 按原顺序（KossJS → Bun → Deno → Node）选择适配器，并调用 `setGlobalAdapter`。  
- 当用户传入 `backend` 时，内置适配器被完全替换，但上层 API 行为保持一致（如 `types`、`pointer`、`array`、`struct` 等与适配器无关，因此不受影响）。  
- 用户后端加载的库句柄类型不受 SenRi 限制，`Library` 类仅将其作为不透明值传递。

## 5. 边界情况与注意事项

| 场景 | 处理方式 |
|------|----------|
| 用户后端 `open` 返回 `null` 或 `undefined` | 允许，但后续 `bind` 会因句柄无效而失败，包装器会捕获并抛出 `FFIError`。 |
| 用户后端 `alloc` 返回的对象缺少 `__buf` | `Pointer` 构造时会检测，若缺失则抛出 `FFIError`。 |
| 用户后端 `registerCallback` 返回的对象缺少 `__cb` | `callback` 函数会检测，若缺失则抛出 `FFIError`，因为无法释放。 |
| 用户后端 `bind` 返回的函数行为不符合类型描述 | 可能导致运行时类型错误，但 SenRi 无法检测，由用户保证。 |
| 用户后端 `bindAsync` 未实现，但 `funcAsync` 被调用 | 降级为同步包装，并输出警告（一次）。 |
| 用户后端抛出非 `Error` 对象 | 包装器会将 `String(e)` 作为消息，构造 `FFIError`。 |
| 多次调用 `Library.load` 切换后端 | 全局适配器会被新后端覆盖，但已存在的 `Library` 实例仍使用其创建时的适配器（通过实例私有字段保存）。**强烈建议**在切换后端前手动释放所有通过旧后端分配的内存和回调，否则旧 `Pointer` 的 `free` 操作会使用新后端的 `free` 方法，可能导致崩溃或内存泄漏。 |
| 用户后端中的 `close`、`free`、`unregisterCallback` 抛出异常 | 包装器静默捕获，仅记录（若可能），不影响上层。 |
| 用户后端未实现 `bindAsync`，但内置适配器有异步实现 | 仅当用户使用自定义后端时，`funcAsync` 使用降级逻辑；内置适配器不受影响。 |
| 多线程环境下同时切换后端 | 当前设计不支持并发切换，请确保在单线程或串行化场景下使用。 |
| 用户后端烟雾测试失败（alloc 或 getErrno 不符） | 在 `Library.load` 阶段即抛出 `FFITypeError`，并指明具体失败原因。 |
| 切换后端时存在活跃资源 | 默认抛出异常阻止；若使用 `force: true`，则输出红色警告并强制切换（风险自负）。 |
| 用户后端提供了 `init` 但抛出异常 | 包装器将异常转换为 `FFIError` 并中止加载过程。 |
| 进程退出时适配器仍有活跃资源 | 资源注册表会在进程退出时被清空（由 Node 等运行时自动回收），但底层内存可能泄漏。用户应确保显式释放。 |

---

## 6. 设计决策说明

- **为什么强制所有内存管理方法？**  
  `Pointer` 和 `struct` 的读写操作都依赖 `alloc/free/addressOf`，若允许缺失，会导致上层 API 大面积失效。

- **为什么 `bindAsync` 是可选？**  
  并非所有 FFI 引擎都提供异步支持（如 Node 的 `koffi` 本身是同步的），降级为同步包装可保证基本可用性，且用户可通过警告知晓。

- **为什么类型映射方法在包装器中要返回序列化字符串？**  
  内核依赖这些方法生成缓存键，返回稳定的字符串能保证缓存正常工作，且不干扰用户后端的 `bind`（后者直接接收 `NormalizedType`）。

- **为什么全局适配器会被替换，而不是支持多实例？**  
  SenRi 顶层函数（`alloc`、`callback`）是模块级单例，它们必须依赖一个全局适配器。若支持多后端，需要将这些函数改为实例方法，设计改动过大。权衡后采用当前方案。

- **为什么线程安全暂未处理？**  
  目标运行时中，FFI 操作通常在主线程进行，且 SenRi 本身未显式设计为多线程安全。我们通过文档明确提示，避免预期外的并发问题。

- **为什么要引入资源注册表和切换保护？**  
  这是对用户常见误操作的防御性设计。实测表明，大多数用户会在切换后端时忘记释放旧资源，从而导致崩溃。通过显式阻止并给出明确错误信息，可以大幅提高安全性，同时为高级用户提供 `force` 选项保持灵活性。

- **为什么烟雾测试只测试 `alloc` 和 `getErrno`？**  
  这两个方法不依赖外部动态库路径，可以在构造时安全调用。其他方法（如 `open`、`bind`）依赖实际路径和符号，无法通用验证。烟雾测试的目标是尽早发现契约违反，而非全面覆盖。

# 补充设计说明：边缘案例防御机制

---

## 问题一：Library 实例与适配器的强绑定 → “僵尸句柄”风险

**场景还原**：  
用户创建 `Library` 实例 A（绑定适配器 X），随后切换全局适配器至 Y。此时实例 A 内部仍持有 X 的引用，但其 `func` 方法在调用时会通过 `this._adapter` 调用 X，而 X 所对应的用户后端可能已被销毁或状态不完整。更危险的是，如果 X 所持有的 OS 级句柄（如 `dlopen` 返回值）在适配器切换时被用户后端的 `destroy` 或 `close` 关闭，那么实例 A 的后续调用将操作无效句柄，导致段错误。

**解决方案：适配器版本戳 + 调用前校验**

在 `Library` 实例中存储两个信息：
- 其创建时的适配器引用（用于实际调用）。
- 其创建时的**全局适配器版本号**（一个单调递增的整数，每次 `setGlobalAdapter` 执行时递增）。

在 `Library.func` 和 `Library.funcAsync` 的调用入口处，增加一道防御性检查：  
比较当前全局适配器的版本号与实例存储的版本号。若不一致，说明全局适配器已被替换，实例处于“僵尸”状态，框架**立即抛出明确的 `FFIError`**，阻止调用，并提示“Library 实例已过期，请重新加载库”。

此机制的关键价值在于：**将隐式的、不可预测的崩溃，转化为显式的、可捕获的异常**。用户不会在段错误中迷失方向，而是收到清晰的指引（“重新加载库”）。

---

## 问题二：FinalizationRegistry 的时机不可控 → 回调释放竞态

**场景还原**：  
`callback` 函数创建了一个回调指针，并依赖 `FinalizationRegistry` 在 `Pointer` 对象被 GC 时自动调用 `unregisterCallback` 释放 C 层回调。但若全局适配器在 GC 发生之前被切换，当 GC 最终触发清理回调时，`getGlobalAdapter()` 返回的将是**新适配器**，而非创建该回调的旧适配器。此时调用新适配器的 `unregisterCallback`，其行为不可预测——可能是什么都不做（泄漏），也可能是尝试释放一个它不管理的指针（崩溃）。

**解决方案：回调资源的显式归属 + 切换时同步清理**

核心思路：**`FinalizationRegistry` 不应是唯一的释放路径，而是最后的防线**。

1. **显式归属标记**：在 `registerCallback` 返回的回调描述对象中，除 `__ptr` 和 `__cb` 外，附加一个 `_adapterId` 字段，记录创建该回调的适配器唯一标识。

2. **适配器切换时的强制清理**：在 `setGlobalAdapter` 执行切换之前，**同步遍历资源注册表中所有已注册的回调描述对象**，对其调用 `_adapterId` 所对应适配器的 `unregisterCallback` 方法，并从注册表中移除。此操作在切换发生前执行，确保旧适配器的回调在切换前已被完全释放。

3. **FinalizationRegistry 的降级角色**：GC 触发的清理回调仍然保留，但不再调用 `getGlobalAdapter()`，而是直接从 `Pointer` 实例中读取存储的 `_adapterId` 和 `_cb`，调用对应的适配器。若该适配器已被销毁，则静默忽略（因为同步清理阶段已经处理过了）。

这一设计将回调释放的**主动权从 GC 转移到了适配器切换流程**，确保了切换的原子性和资源完整性。

---

## 问题三：烟雾测试调用 alloc(1) 的副作用风险

**场景还原**：  
烟雾测试在 `Library.load` 阶段调用 `alloc(1)` 并立即 `free`，目的是验证用户后端的 `alloc` 方法返回值形状。但某些后端可能：
- 对分配大小有最小对齐要求（例如总是分配 16 字节，即使请求 1 字节）。
- 在首次分配时执行昂贵的初始化（如建立内存池、映射页表）。
- 将 `alloc` 操作记录到日志中，产生噪音。

这些副作用虽然不影响正确性，但在性能敏感或对后端行为有严格观测需求的环境中，可能是不受欢迎的。

**解决方案：烟雾测试的可配置化**

将烟雾测试设计为**非强制**，通过环境变量或 `Library.load` 的选项参数控制：

- **默认启用**（保证契约验证的安全性）。
- 用户可通过 `SENRI_FFI_SKIP_SMOKE_TEST=1` 环境变量跳过测试。
- 或者在 `Library.load` 的第二个参数扩展为对象形式（当需要传递更多选项时），其中包含 `skipSmokeTest: true` 选项。

跳过烟雾测试的用户需自行承担契约违反的风险——但正如文档开篇所述，自定义后端功能本身即为高阶能力，用户理应知晓自己在做什么。

---

## 问题四：createBackendWithFallback 的回退链限制

**场景还原**：  
当前设计仅支持“部分实现 → 内置适配器”的单层回退。但一个更复杂的场景是：用户希望用后端 A 覆盖 `alloc` 和 `free`，用后端 B 覆盖 `bind` 和 `bindAsync`，其余方法回退到内置适配器。这是一个“多层委托”的需求。

**解决方案：明确设计边界 + 声明可组合性**

首先在文档中明确：当前版本仅支持单层回退，即部分实现 + 内置适配器。这是有意为之的简化设计，避免引入复杂的委托链调度开销和调试难度。

同时，在接口层面提供一个声明性的扩展点：`createBackendWithFallback` 的第二个参数类型并非限定为“内置适配器”，而是“任意满足 `LibraryLike` 接口的对象”。这意味着用户理论上可以构造一个“中间后端”（由另一个自定义后端和内置适配器组合而成），然后将其作为回退目标传入。

形式上：
- 用户先调用 `createBackendWithFallback(partialA, builtin)` 得到一个后端 B。
- 再调用 `createBackendWithFallback(partialB, B)` 得到一个后端 C。
- 这本质上是**委托链的组合**，虽然每一步都是单层回退，但通过连续组合可以实现多层效果。

文档应在“设计决策说明”中明确此模式，并提示用户注意委托链过长可能带来的性能开销和调试复杂度。

## 整体机制总结

| 问题 | 防御机制 | 触发时机 |
| :--- | :--- | :--- |
| 僵尸句柄跨后端调用 | 适配器版本戳 + 实例方法入口校验 | 每次 `func` / `funcAsync` 调用 |
| 回调释放 GC 竞态 | 切换时同步清理 + 显式归属标记 | `setGlobalAdapter` 执行前 |
| 烟雾测试副作用 | 环境变量/选项控制跳过 | `Library.load` 阶段 |
| 回退链深度限制 | 文档明确定义 + 支持递归组合 | 设计阶段 |

这些机制共同构成了自定义后端特性的**可靠性保障层**，使其从“可用”提升到“在复杂生产环境中可依赖”。
